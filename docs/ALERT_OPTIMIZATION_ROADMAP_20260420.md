## ✅ P1 完成总结（高点缓存重建）

**为什么要做**：旧 `historicalHighCacheV2` 覆盖不全（仅 160 symbol）、数据陈旧（4+ 天不更新）、时间框架不贴交易（1w/1m/6m/1y/all 语义模糊、`all` 受限合约上线日）、存储格式笨（单个 JSON 全量写）。

**实际交付**：
- 新表 `historical_highs(symbol, timeframe, high_price, high_at, window_start, window_end, collected_at)`，主键 `(symbol, timeframe)`
- **活跃 symbol 清单**：从 `exchangeInfo` + 过滤（非 TRADING、USDC/季度/下架、`isRiskyToken`）+ 双阈值死币过滤后约 **469 个**
- **死币过滤双阈值（hotfix）**：
  - 24h 成交额 < 1M USDT：直接剔除
  - 成交额在 [1M, 50M) 区间：要求 24h 振幅 >= 3%
  - 成交额 >= 50M：跳过振幅检查（主流币低波动也算活跃；首次部署时 BTC/ETH/BNB 被错误剔除的 bug 由此修正）
- **三层刷新**：
  - 冷层 24h：遍历全部活跃 symbol，重算 5 档，顺便 prune 死币
  - 温层 1h：遍历 cached symbol，只比对最新 1h K 线是否破高
  - 热层：实时价每次查询时从 `binanceClient.getFuturesPrice` 拉
- **启动非阻塞**：首次冷刷新后台跑，不挡住 app.ts 启动；冷完成耗时约 4 分钟
- **ATH 定义**：只用合约 1d K 线，从 `onboardDate` 到今天取 max high；**不引入 spot**
- **查询 API**：`queryHigh(symbol, tf)` / `queryAllHighs(symbol)` / `getRankingByProximityToHigh(tf, limit)`（用实时价计算距离，不从缓存拿当前价）

**引用点改造**：
- `bot.ts` 的 `/high` `/near_high` 完全委托给 `HighCommandHandler`（删 ~170 行重复代码）；`/cache_status` 换 v3 字段；`/cache_update` 触发 `runColdRefresh()`
- `BreakthroughDetectionService`（既有用户警报）切到 v3 API + 加旧 timeframe (`1w/1m/6m/1y/all`) → 新 (`7d/30d/180d/52w/ATH`) 映射，不破坏已存在的用户警报数据

**验证**：
- 18 个测试全过（12 model + 6 service 集成）
- 服务器 DB 当前 2345 行 / 469 symbols / 5 档齐全
- BTC ATH = 126208.50 (2025-10-06)，ETH ATH = 4957.67 (2025-08-24)，主流币正确入库

## ✅ P2 完成总结（突破报警服务）

**实现概要**：
- 每 10 分钟扫一次全市场活跃 symbol
- 流程：批量拉 24h ticker → 对每个 cached symbol 按档位优先级 (`ATH > 52w > 180d > 30d > 7d`) 找**最高破档**（一个 symbol 一次扫只触发最高那一档，避免重复）→ 对候选拉 21 根 1h K 线做二次确认 → 通过 + 未冷却就发
- **档位映射**：`L3_weak(7d) / L2_mid(30d) / L1_strong(180d) / L1_extreme(52w 或 ATH)`
- **二次确认**（纯函数，单元测试覆盖）：
  - 放量：最新 1h `quoteAssetVolume` >= 前 20 根 1h 均量 × **1.5**
  - 持续：最新 1h 的 `low` > 参考高点（本小时还没跌回）
  - 不刚上市：合约年龄 >= 档位窗口（ATH 档不限）
- **冷却**：同 symbol + 同 tier **6 小时**内不重推；升档 (30d → 180d) 立即再推
- **推送**：Telegram + ESP32 `breakthrough` 类型（已有，复用）
- **默认关闭**，发 `/breakout_on` 或菜单 toggle 激活

**新增**：
- 表 `breakout_alerts` + `breakout_alert_config`（开关）
- 命令 `/breakout_status` `/breakout_on` `/breakout_off`
- 菜单主页第 4 个 toggle「🚀 突破报警」+ 状态页突破段（含今日档位统计）

**测试**：17 个新测试（10 model + 7 service 纯逻辑）

## 📌 起点：当前系统的业务盲点

基于 4/19 一天 450 条推送的分析，核心问题有三个：

1. **价格警报"凡满足即推"**—— 一波行情 39 个币同时涨 10%，用户被淹没
2. **信号只有触发/不触发二值**，没有"强信号"和"弱信号"的分层
3. **没有结构性突破识别**（52w / ATH / 30d 新高），这是右侧动量最稀缺的信号

对应三个方案（#2 Top-N、#3 共振分级、#4 突破）。其中 #4 有个前置基础设施要先解决。

## 🗄️ 前置工作：高点缓存库重建

### 现状问题

代码文件：`src/services/historicalHighCacheV2.ts`，数据存 `data/historical-high-cache.json`。

| 问题 | 现状 | 影响 |
|---|---|---|
| **覆盖不全**（部分 intentional） | 只缓存 160 个 symbol | 新上市币 + 活跃币可能缺失；但低流动性/低波动"死币"不在缓存中是**合理的**（浪费资源） |
| **更新不及时** | 最后更新 2026-04-16（4 天前），靠手动触发 | 查询的"高点"和"当前价"都过期 |
| **时间框架不贴交易** | 只有 1w/1m/6m/1y/all | 缺关键的 30d/52w/ATH；`all` 命名模糊 |
| **存储格式笨重** | 整个缓存序列化成单个 JSON（234KB，全量写） | 并发更新冲突；按 symbol 增量更新要重写整个文件 |
| **无持续更新任务** | 有 `triggerManualUpdate` 但没定时调度 | 用户查询常拿到几天前的数据 |

### 重建方案

**1) 存储换 SQLite**  
新建表 `historical_highs(symbol, timeframe, high_price, high_at, window_start, window_end, collected_at)`，主键 `(symbol, timeframe)`。好处：
- 按 symbol 增量更新不用全量重写
- 并行 worker 可更新不同 symbol
- 跟现有 `crypto-alerts.db` 共库，不引新依赖

**2) 时间框架重新定义（币圈节奏，不照搬美股）**  
只保留 5 档：

| 键 | 含义 | K 线粒度 |
|---|---|---|
| `7d` | 7 天 / 一周（战术） | 1h |
| `30d` | 30 天 / 一个月（短期趋势） | 1d |
| `180d` | 180 天 / 半年（中期趋势） | 1d |
| `52w` | 52 周 / 一年（长期压制位） | 1d |
| `ATH` | 合约上线后历史最高（结构性） | 1d |

放弃 `1w / 1m / 6m / 1y / all`（语义模糊），不引入 `60d` / `90d`（跟 52w/180d 重复度高）。

**3) ATH 的定义：只用合约数据，不引入 spot**  
ATH = 该 futures 合约从上线首日至今的最高收盘价。上线日期从 Binance `exchangeInfo` 拿 `onboardDate`。**不引入现货 spot 数据**——避免"现货 ATH vs 合约价"两个维度混淆（永续是合约交易，触发做多做空用的是合约行情，ATH 就看合约本身）。

**4) 三层刷新节奏**

| 层 | 数据 | 频率 | 做法 |
|---|---|---|---|
| **冷层** | 历史 K 线 | 每日 1 次（凌晨 4 点 UTC+8） | 拉所有 timeframe 窗口，重算高点 |
| **温层** | 当日新高检查 | 每小时 1 次 | 只查最近 1h K 线，比对缓存高点，命中则更新 |
| **热层** | 实时当前价 | 每次查询 | 不入缓存，直接从 `binanceClient.getFuturesPrice` 拉 |

**关键洞察**：缓存的是"**过去的高点**"（变化慢），不是"**当前价**"（变化快）。当前逻辑混在一起才导致 4 天没更新时查询结果失真。

**5) 覆盖范围：自动发现 + 死币过滤**  
每次冷层刷新前拉 `GET /fapi/v1/exchangeInfo` 拿到全量永续合约清单，自动把新上市的加入缓存。同时**主动剔除死币**：

| 过滤条件 | 阈值（可调） | 说明 |
|---|---|---|
| 24h 成交额 | < 1M USDT | 长期流动性不足 |
| 7d 最大振幅 | < 5% | 波动太小没交易价值 |
| 合约状态 | 非 `TRADING` | 已下架 / 挂起 |
| 已命中系统风险名单 | `isRiskyToken(symbol)` | 复用现有过滤器 |

每日冷层扫描时重算一遍"活币清单"，死了的从缓存删、活过来的加回来。预期缓存 symbol 数约 200-300 个（而不是现在的 160 固定或 500+ 全量）。

**6) API 改造**
- `queryHighs(symbol) → { '7d': {price, at}, '30d': ..., 'ATH': ... }` 一次返回所有 timeframe
- `distanceToHigh(symbol, timeframe) → { distancePct, gainNeededPct }` 用实时价当场算，不从缓存拿当前价

## 📋 方案 #2：Top-N 排序推送

### 核心思想

**把"触发"和"推送"解耦**，中间加一层"排序 + 取 Top N"。

### 现状问题

价格警报 `config=4`（1h 涨幅 ≥10% 全市场）在 4/19 触发 212 次，涉及 39 个币；其中 HIGHUSDT 单独就被推了 20 次。本质是"**把筛选工作留给用户**"。

### 新逻辑

**Step 1 候选池**：扫描全市场，把所有满足阈值的 symbol 放入候选池（不立即推）。

**Step 2 打强度分数**。简化版（建议先用）：

```
分数 = max(1h 涨幅%, 15m 涨幅%) × 1.0
     + 1h OI 变化% × 0.8
     + (当前小时量 / 24h 均量) × 10
     + (是否同时负费率) × 5
```

权重可调，先全用 1.0，观察一周再校准。

**Step 3 取 Top N**（N=3）
- 按分数降序取前 N
- 超出的汇总成一条消息："此外还有 X 个币满足条件，Top K: AAA / BBB / CCC"

**Step 4 同币冷却**  
Top N 推过的币，2 小时内不再进候选池（避免同一波行情反复占用名额）。

**Step 5 自我观察**  
每条推送附"此币当轮排名 #N / 候选池 M 个"，让用户判断是真稀缺还是池子本来就小。

### 为什么不会漏大涨币

你担心"同币冷却会漏掉加速上涨"—— Top N 排序不会漏：
- 加速上涨 = 分数继续最高 = 还是 Top N
- 慢涨耗尽的币会被更急的币挤掉

### 预期收益

从 212 条/天 → 约 40 条/天，减 80% 噪音，且每条都是"当轮最强"。

### 实施范围

- 改造目标：价格警报 `config=4` + funding negative 触发
- 不动 `potential_alerts`（它本来就是排序后的）

## 📊 方案 #3：多指标共振分级推送

### 核心思想

**报警不是"触发/不触发"二值，而是"信号强度"多档**。

### 三档定义

| 档位 | 定义 | 推送方式 |
|---|---|---|
| **L1 稀缺** | 三维度及以上同向共振 | ESP32 语音 + Telegram（重要打扰） |
| **L2 一般** | 两维度共振 | 仅 Telegram（普通打扰） |
| **L3 噪音** | 单维度触发 | 默认静默，每日汇总成一条日报 |

### 维度清单（对每个 symbol 实时打 6 个维度）

| 维度 | 信号含义 | 触发条件举例 |
|---|---|---|
| 📈 **价** | 短期涨幅 | 1h 涨 ≥ 10% |
| 📊 **OI** | 持仓量变化 | 1h OI 涨 ≥ 15% |
| 💸 **费率** | 费率状态 | 负费率 或 费率急剧变化 |
| 🔊 **量** | 成交放大 | 当前小时量 ≥ 24h 均量 × 2 |
| 🎯 **突破** | 结构信号 | 突破 30d 高点（需 #4） |
| 💰 **资金流** | 大单异动 | 单笔大单频率上升（需额外数据源） |

### 评级逻辑

每个 symbol 依次检查 6 个维度是否**同向满足**，计数共振维度数：
- 共振 ≥ 3 → **L1**
- 共振 = 2 → **L2**
- 共振 = 1 → **L3**
- 0 共振 → 不推

**同向**= 都是看涨 / 都是看跌。价涨 + OI 涨 + 负费率 = 3 维看涨 = L1；价涨 + OI 跌 = 背离，0 共振不推。

### 与方案 #2 的关系

**互补不冲突**：
- #3 管 "**这条要不要推、推多吵**"
- #2 管 "**同一档位里先推谁**"
- 流程：扫描 → 打分 → 分档 → 每档内做 Top N

### 第一版简化

不实现"量"和"资金流"（量能要接 K 线历史，资金流要挂 orderbook 流）。
只实现"**价 + OI + 费率 + 突破**"四维，共振计数从 4 里数，阈值仍 L1≥3 / L2=2 / L3=1。

### 与现有 potential_alerts 的整合

`potential_alerts` 本质已是 L1（价 + OI + 费率 三共振）。改造后应**并入统一评级系统**，不跑两套独立逻辑。

## 🎯 方案 #4：突破信号（依赖高点缓存）

### 触发档位（递进）

| 档位 | 触发 | 备注 |
|---|---|---|
| **L3 弱** | 突破 7d 高点 | 本周新高，信号密但价值低 |
| **L2 中** | 突破 30d 高点 | 一个月新高，值得关注 |
| **L1 强** | 突破 180d 高点 | 半年新高，中期压制位被突破 |
| **L1 极** | 突破 52w / ATH | 年度 / 历史新高，结构性突破 |

所有档位都需要**二次确认**（见下）。

### 二次确认（过滤假突破）

- **放量**：突破当根 K 线量 ≥ 最近 20 根 K 线均量 × 1.5
- **持续**：突破后 30 分钟内没跌回原高点下方（避免"戳一下就回"）
- **不刚上市**：symbol 上市时间 ≥ 对应窗口（突破 30d 高点要求至少存在 30 天）

### 实现形态

独立 service（如 `BreakoutAlertService`），每 10 分钟扫一次：

1. 遍历所有 symbol 当前价（走死币过滤后的清单，和高点缓存一致）
2. 依次对比 `7d / 30d / 180d / 52w / ATH` 五档缓存高点
3. 命中高点 → 检查放量和持续
4. 通过 → 作为"突破"维度进入 #3 评级，跟其他维度共振计分
5. 同时其他维度共振 → 升级为 L1

### 冷却

同 symbol、同档位 6h 内不重推。升档（L2 → L1）立即推（升档是新信息）。
